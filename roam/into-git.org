#+title: Into-Git

** Git

Local Git Directory Layout

When you run git init in a folder:

A hidden .git/ directory is created.
It contains all the version history, branches, tags, and staging data.
If .git/ is deleted, the folder is no longer a Git repo.

** Status

A file can be in one of several states in a Git repository. Here are a few important ones:

untracked: Not being tracked by Git
staged: Marked for inclusion in the next commit
committed: Saved to the repository's history

The git status command shows you the current state of your repo. It will tell you which files are untracked, staged, and committed.

** Staging

The contents.md file has been created, but as we saw, it's untracked. We need to stage it (add it to the "index") with the git add command before committing it later.

Without staging, no files are included in the commit—only the files you explicitly git add will be committed.

Here's the command:

~git add <path-to-file | pattern>~

** Commit

After staging a file, we can commit it.

A commit is a snapshot of the repository at a given point in time. It's a way to save the state of the repository, and it's how Git keeps track of changes to the project. A commit comes with a message that describes the changes made in the commit.

Here's how to commit all of your staged files:

~git commit -m "your message here"~

*** What happens when you commit?
Git takes a snapshot of the file,
Stores it as a blob object,
Records a tree object representing directory structure,
Creates a commit object pointing to the tree + metadata (author, timestamp, message).


** Git Log

A Git repo is a (potentially very long) list of commits, where each commit represents the full state of the repository at a given point in time.

The git log command shows a history of the commits in a repository. This is what makes Git a version control system. You can see:

    Who made a commit
    When the commit was made
    What was changed

~git --no-pager log -n 10~
or
~git --no-pager log -n 10 --online --parents --graph~

A Commit Hash

Each commit has a unique identifier called a "commit hash". This is a long string of characters that uniquely identifies the commit. Here's an example of mine:

~5ba786fcc93e8092831c01e71444b9baa2228a4f~

For convenience, you can refer to any commit or change within Git by using the first 7 characters of its hash. For mine, that's 5ba786f.

You may have noticed that even though we (you and I) both have the same content in our repositories, we have different commit hashes. While commit hashes are derived from their content changes, there's also some other stuff that affects the end hash. For example:

    The commit message
    The author's name and email
    The date and time
    Parent (previous) commit hashes

All this to say that hashes are effectively unique in practice. While SHA-1 collisions are possible under contrived conditions, you won't accidentally create two different commits with the same hash, and because they're generated automatically for you, you don't need to worry too much about what goes into them right now.
Note: Hash = SHA

Git uses a cryptographic hash function called SHA-1 to generate commit hashes. We won't go into the details of how SHA-1 works in this course, but it's important to know because you might also hear commit hashes referred to as "SHAs".

** Commit Quality
Commits should contain changes related to a **single topic** to maintain clarity and ease of understanding for both current and future collaborators.
- **Selective Staging:** The Git staging area enables adding **specific files or even parts of files** (`git add -p`) to ensure commits stay focused.
- **Commit Message Best Practices:**
  - Keep the **subject line concise**, ideally under 80 characters.
  - Use the commit message body to answer:
    - *What changed?*
    - *Why was the change made?*
    - *Are there any notable implications or points to watch out for?*
  - A clear message aids code review and future maintenance.:

#### Branching Strategies
- Git provides tools but **does not enforce branching conventions**; teams must define and document their own strategies.
- Branch conventions help avoid conflicts, improve project organization, and assist onboarding.
- Two extreme workflow models:
  - **Always Be Integrating (ABI):** A single branch where all commits are integrated frequently. Requires small commits and strong testing/QA to maintain quality.
  - **Multiple Branch Types:** Separate branches for features, releases, experiments, and different deployment states (e.g., develop, staging, production). More complex but flexible.
- **Long-running branches:** Exist throughout the project lifecycle (e.g., `main`, `develop`, `staging`). Direct commits are usually avoided; changes come through merges or rebases.
- **Short-lived branches:** Created for specific tasks like features, bug fixes, or experiments; deleted after integration.

#### Popular Branching Workflows
| Workflow    | Description                                                                                           | Branch Types                         |
|-------------|---------------------------------------------------------------------------------------------------|------------------------------------|
| GitHub Flow | Simple, single long-running branch (`main`), all work done in short-lived branches for features or fixes. | 1 long-running (`main`), many short-lived |
| Git Flow    | Structured with `main`, `develop`, release, and feature branches. Supports release management and QA. | Multiple long-running and short-lived branches |

- Tools like Tower GUI can simplify following branching workflows by automating common tasks.

#### Pull Requests (PRs)
- PRs are **not a core Git feature** but provided by hosting platforms (GitHub, GitLab, Bitbucket).
- Purpose:
  - Facilitate **code review and collaboration** by inviting feedback before merging.
  - Enable contributions to repositories where you have no direct write access (via forks).
- Workflow example:
  - Fork a repository.
  - Create a feature branch.
  - Make and commit changes.
  - Push branch to your fork.
  - Open a PR to propose merging changes back to the original repository.
- PR interfaces vary by platform but share the same principles.

#### Merge Conflicts
- Occur during integration actions like merge, rebase, cherry-pick, pull, or stash application.
- Typical cause: **Contradictory changes**, e.g., same line altered differently or file modified in one branch and deleted in another.
- Git detects conflicts and **halts the operation**, requiring manual resolution.
- Conflict markers in files:
  - `<<<<<<< HEAD` - current branch changes
  - `=======` - separator
  - `>>>>>>> other-branch` - incoming branch changes
- Resolution:
  - Manually edit files to choose or combine changes.
  - Use GUI tools (e.g., Tower) or dedicated merge tools for visualization and easier conflict resolution.
- You can **abort merge or rebase conflicts** anytime using commands like `git merge --abort` or `git rebase --abort`.
- After resolving conflicts, commit the changes to complete the integration.

#### Integrating Branches: Merge vs. Rebase
| Feature          | Merge                                    | Rebase                                  |
|------------------|------------------------------------------|----------------------------------------|
| Resulting History| Creates a **merge commit** connecting branches, retaining branch structure | Rewrites commit history to create a **linear history** without merge commits |
| Use Case         | Common for shared branches, preserves context of branch topology | Ideal for cleaning local commit history before integration |
| Risks            | Can clutter history with merge commits  | Rewriting history can cause issues if commits are already pushed/shared |
| Command          | `git merge <branch>`                     | `git rebase <branch>`                   |

- **Important rule:** Avoid rebasing commits that have been pushed to shared repositories to prevent conflicts for collaborators.

### Core Concepts Explained

- **Git staging area:** Buffer zone to choose exactly which changes to include in the next commit.
- **Long-running branches:** Persistent branches representing main development or deployment stages.
- **Short-lived branches:** Temporary branches for isolated work, deleted after integration.
- **Pull requests:** Mechanism for code review and collaboration on Git hosting platforms.
- **Merge conflict:** Occurs when two branches have incompatible changes that Git cannot automatically reconcile.
- **Merge commit:** A commit automatically created by Git to join two divergent branches.
- **Rebase:** Reapplies commits on top of another branch to create a linear history, rewriting commit hashes.
